# Derived-view rendering â€” operator visibility model

<!-- Last updated: 2026-05-06 (Spec 254 â€” Approach D Phase 1) -->

FORGE's tracking views (backlog, changelog, spec index) are **derived** from per-spec sources, not stored as canonical files. This guide explains how to read project state and what each command surface returns.

## Why derived views

Before Spec 254, `docs/backlog.md`, `docs/specs/CHANGELOG.md`, and `docs/specs/README.md` were single files modified by every `/implement` and `/close`. Concurrent close operations across multiple Claude Code tabs (or developers) produced near-guaranteed merge conflicts on these files.

Approach D eliminates the conflict surface entirely: there is no single shared file to conflict on. The "canonical" views are computed on demand from:

- **Spec frontmatter** under `docs/specs/NNN-*.md` (status, priority score, dependencies, etc.) â€” already authoritative per [ADR-028](../decisions/ADR-028-single-source-of-truth-for-spec-status.md).
- **Append-only per-spec event streams** under `.forge/state/events/<spec-id>/<event-type>.jsonl` â€” conflict-free by construction (each spec writes only to its own subdirectory).

## Renderer trio

Three renderers live under `.forge/lib/`:

| Renderer | Output | Source |
|----------|--------|--------|
| `render_backlog.py` | Ranked backlog table | spec frontmatter |
| `render_changelog.py` | Chronological event log | per-spec event streams (+ frontmatter fallback) |
| `render_spec_index.py` | Spec index | spec frontmatter |

All three are unified Python implementations (per [ADR-091](../decisions/ADR-091-context-engineering-for-slash-commands.md) and the `strategic-scope.py` precedent). Determinism â€” the central correctness guarantee â€” is verified by running each renderer twice and asserting byte-identical stdout.

```bash
python3 .forge/lib/render_backlog.py        # writes to stdout
python3 .forge/lib/render_changelog.py
python3 .forge/lib/render_spec_index.py
python3 .forge/lib/render_backlog.py --output docs/backlog.md
```

## Mode choice: `delete` vs `generated` vs `skip-canonical` vs `split-file`

When a project migrates to Approach D, the canonical files (`docs/backlog.md`, `docs/specs/CHANGELOG.md`, `docs/specs/README.md`) become one of:

- **`--mode=delete`**: removed entirely. `cat docs/backlog.md` returns "No such file." Operators read state via `/matrix` or by invoking renderers directly. Keeps the working tree minimal.
- **`--mode=generated`** (Phase 1 default): retained as artifacts with a `# DO NOT EDIT â€” generated` header. Regenerated at `/matrix` time. Preserves `cat`-affordance + GitHub-rendered backlog page + git-historical-diff for committed snapshots.
- **`--mode=skip-canonical`**: leaves canonical files untouched (operator-curated). Renderers exist but `/matrix` does not write them. Useful as a staged Phase 1 rollout.
- **`--mode=split-file`** (Spec 398, recommended for projects with operator-curated content): canonical file stays operator-owned; renderer writes a separate generated artifact under `docs/.generated/`; the curated parent references the generated artifact through a one-line include marker.

## Split-file architecture (Spec 398)

The split-file mode resolves the failure-mode class around mixed-authority single files (sentinel-region fragility, lost edits, CRLF/BOM corruption on Windows) by giving each side a clean authority boundary.

### Authority boundary

| Side | Path | Author | What it contains |
|------|------|--------|------------------|
| Curated parent | `docs/backlog.md`, `docs/specs/CHANGELOG.md`, `docs/specs/README.md` | Operator | Free-form narrative, history, sprint plans, deferred-scope tables, plus a one-line include marker |
| Generated artifact | `docs/.generated/backlog-table.md`, `changelog-entries.md`, `spec-index-table.md` | Renderer | Rendered output only; first line is `<!-- DO NOT EDIT â€” generated by .forge/lib/render-*.py â€” see docs/process-kit/derived-view-rendering-guide.md -->` |

The renderer **never** opens, reads, or mutates the curated parent. The assembler reads both but writes neither â€” its output goes only to stdout.

### Include-marker syntax

A single line in the curated parent:

```
<!-- FORGE-INCLUDE: <relative-path-from-curated-parent> -->
```

- Path is **relative to the directory of the curated parent**.
- Whitespace tolerated as `<!-- FORGE-INCLUDE:<one-or-more-spaces><path><one-or-more-spaces>-->`.
- Multiple include markers per curated parent are permitted.
- Case-sensitive; single-line; line-anchored.

Worked examples:

| Curated parent | Marker |
|----------------|--------|
| `docs/backlog.md` | `<!-- FORGE-INCLUDE: .generated/backlog-table.md -->` |
| `docs/specs/CHANGELOG.md` | `<!-- FORGE-INCLUDE: ../.generated/changelog-entries.md -->` |
| `docs/specs/README.md` | `<!-- FORGE-INCLUDE: ../.generated/spec-index-table.md -->` |

### Assembler invocation

The assembler is a render-time projection. It never persists output.

```bash
# Interactive view at /matrix time (called automatically when docs/.generated/ exists)
python .forge/lib/assemble_view.py docs/backlog.md

# Pipe to a pager:
python .forge/lib/assemble_view.py docs/backlog.md | less
```

Exit codes: `0` (success â€” may include stderr stale-warning), `1` (referenced generated artifact missing â€” error names the path), `2` (argument or I/O error).

A stderr-only stale warning (`stale view: run /matrix to regenerate`) appears when any spec frontmatter or event-stream file mtime is newer than the generated artifact's mtime. The warning is advisory; it does not fail.

### `cat` vs `git show` behavior

| Command | What it returns under split-file |
|---------|----------------------------------|
| `cat docs/backlog.md` | Curated parent + include marker (NOT the assembled view) |
| `python .forge/lib/assemble_view.py docs/backlog.md` | Assembled view (curated parent with marker replaced by generated artifact contents) |
| `git show <SHA>:docs/backlog.md` | Curated parent at that revision only â€” the generated artifact lives at `docs/.generated/backlog-table.md` and must be retrieved separately. To reconstruct the historical assembled view, `git checkout <SHA>` and re-run the assembler. Documented as a known limitation. |

### Operator decision tree (which mode for which project state)

- **Greenfield project, no existing curated content**: use `--mode=generated` (simplest â€” single canonical file).
- **Existing project with operator-curated narrative in `docs/backlog.md`** (FORGE-self, mature consumer projects): use `--mode=split-file` â€” operator content survives, renderer output gains burn-in evidence.
- **Lean / experimental project, no human readers of canonical files**: `--mode=delete` is fine.
- **Staged Phase 1 rollout (don't touch canonical yet)**: `--mode=skip-canonical`.

Default for new migrations remains `generated` for backward compatibility; `split-file` is opt-in via `python scripts/migrate-to-derived-view.py --mode=split-file`.

Switch modes later via `python3 scripts/migrate-to-derived-view.py --mode <new-mode>` (re-run is idempotent).

## When canonical files are absent (or generation-only)

The mental model for FOUR cases:

### Case 1 â€” In-progress spec

Spec 254 is at `Status: in-progress`. You want to know its score, dependencies, current step.

| Surface | Returns |
|---------|---------|
| `/matrix` | Renders the backlog table; spec 254 appears with `â†’` rank marker (in-progress) |
| `cat docs/backlog.md` (mode=`generated`) | Same content as `/matrix`'s table â€” last regen of the file |
| `cat docs/backlog.md` (mode=`delete`) | "No such file" â€” invoke renderer instead |
| `git show <SHA>:docs/backlog.md` | Pre-Spec-254 monolithic backlog as it existed at that commit (history is recoverable for either mode) |
| Direct: `python3 .forge/lib/render_backlog.py` | Always works â€” generates from current spec frontmatter |

### Case 2 â€” Closed spec

Spec 390 closed yesterday.

| Surface | Returns |
|---------|---------|
| `/matrix` | Backlog table shows spec 390 with âś… marker |
| `cat docs/backlog.md` (mode=`generated`) | Same |
| `cat .forge/state/events/390/spec-closed.jsonl` | Single JSONL line with `event_type: spec-closed`, timestamp, payload |
| `git log -- docs/specs/390-*.md` | Full revision history of the spec file itself |

### Case 3 â€” Deleted spec

A spec that was deprecated and the spec file was removed.

| Surface | Returns |
|---------|---------|
| `/matrix` | Spec doesn't appear (no frontmatter source) |
| Event streams | Last events for that spec_id remain in `.forge/state/events/<spec-id>/` â€” historical-only |
| `git log -- docs/specs/<deleted-id>-*.md` | Pre-deletion history |

The renderer uses **frontmatter as the authoritative present-state source** and **event streams as the authoritative history source**. Deleting a spec removes it from the present-state view but not from history.

### Case 4 â€” File under both modes (a) and (b)

| Mode | `cat docs/backlog.md` | `git diff` after `/close` |
|------|------------------------|----------------------------|
| `delete` | "No such file" | No diff to canonical files; only event-stream additions in `.forge/state/events/<spec-id>/` |
| `generated` | Shows last regenerated table | Diff includes the regenerated canonical file + event-stream additions |

**Mode `generated` is recommended** when third-party tooling, GitHub rendering, or operators reading from `cat` are part of the workflow. Mode `delete` is appropriate for projects committed to invoking renderers directly.

## Concurrency story (the core motivation)

Two operators run `/close` simultaneously on different specs (say 380 and 381):

1. Each writes to its own `.forge/state/events/<spec-id>/spec-closed.jsonl` â€” **disjoint paths, no conflict**.
2. Each updates its own spec file's frontmatter (`Status: closed`, `Closed: <date>`) â€” **disjoint paths, no conflict**.
3. Mode `generated`: each `/close` regenerates `docs/backlog.md` post-status-change. Last writer wins, but the content is **derived from frontmatter** so both updates are reflected (the regen reads both 380 and 381's frontmatter regardless of which `/close` ran the regen).
4. Mode `delete`: no canonical file write at all. Pure event-stream + frontmatter operation.

Either way: **structurally impossible to produce a merge conflict** on a shared tracking file in the canonical sense.

## Third-party tools

Tools that previously read `docs/backlog.md` directly (CI scripts, dashboards, Claude Code commands) need to either:

1. Invoke the renderer: `python3 .forge/lib/render_backlog.py --no-header` and parse stdout.
2. Read the canonical file when in mode `generated` (works for most cases; expect "No such file" handling for mode `delete`).
3. Read spec frontmatter directly via `.forge/lib/spec_frontmatter.py` for programmatic use.

Example: a CI script that wants the count of open drafts:

```bash
python3 -c "
import sys; sys.path.insert(0, '.forge/lib')
from spec_frontmatter import iter_spec_files, parse_spec_file
from pathlib import Path
n = sum(1 for f in iter_spec_files(Path('docs/specs'))
        if (parse_spec_file(f) or {}).get('status', '').strip() == 'draft')
print(n)
"
```

## Performance budget

Renderers must complete in under 2 seconds for a 500-spec project on commodity hardware (8 GB RAM, SSD). The fixture at `tests/test-spec-254-renderer-perf-500spec.sh` exercises this with a seeded corpus; CI gate fails the build on overrun.

For interactive `/matrix` invocations, the wall-clock target is "feels instant" (< 200ms typical for the FORGE-self project at ~400 specs).

## Event-stream schema

Per-spec, append-only JSONL files at `.forge/state/events/<spec-id>/<event-type>.jsonl`. Each line:

```json
{"timestamp": "2026-05-06T13:42:00Z", "event_type": "spec-closed", "payload": {"message": "Closed via /close (Chat mode)"}}
```

Defined event types (initial set; **forward-extensible** â€” new event types append, no breaking changes):

- `spec-started` â€” `/implement` entry
- `spec-approved` â€” inline approval at `/implement`
- `spec-implemented` â€” `/implement` exit
- `spec-closed` â€” `/close` exit
- `spec-deferred` â€” `/revise` defer disposition
- `spec-deprecated` â€” `/revise` deprecate disposition
- `consensus` â€” one entry per `/consensus` round (round number in payload)
- `revise` â€” `/revise` mutations
- `signal-reference` â€” cross-references to signals captured during the spec

Renderers tolerate unknown event types (skip with stderr warn; never crash). Schema additions to `payload` are forward-compatible.

See `.forge/lib/events.py` for the canonical helpers (`append_event`, `load_events`, `iter_spec_dirs`).

## Command interactions (Spec 399)

Spec 399 introduces the shared **state helper** `.forge/lib/derived_state.py` â€” a single Python module that concentrates ALL schema knowledge for derived state. Slash commands invoke this helper instead of opening canonical files (`docs/backlog.md`, `docs/specs/README.md`, `docs/specs/CHANGELOG.md`) directly. The helper IS the schema contract.

### Why a helper module

Before Spec 399, nine commands read canonical state from these three files using inline file-reads + ad-hoc parsing. Spec 398 added a fourth rendering mode (`split-file`) where the canonical files contain only an include marker â€” direct reads in that mode return narrative + marker, not data. Adding mode-detection + path selection to nine commands creates a 9-way coupling problem; one schema change touches every command.

The helper-as-contract reframe (Candidate B from the round-3 consensus showdown) collapses the 9-way coupling to a single seam: commands depend on the helper's Python API and CLI signature; the helper internalizes mode-detection + parsing + fallback. Future schema evolution updates the helper + its unit tests + the renderers; no command edits.

### Public Python API

- `detect_mode(project_root) -> "split-file" | "generated" | "skip-canonical"`
- `get_backlog_rows(project_root) -> list[dict]` â€” ranked backlog
- `get_spec_index(project_root) -> list[dict]` â€” spec index
- `get_changelog_entries(project_root) -> list[dict]` â€” newest-first
- `should_skip_canonical_write(project_root) -> bool` â€” write-side gate

### CLI signature

Commands invoke the helper via the Spec 401 wrapper â€” single byte-identical invocation across platforms and all 4 mirrors:

`.forge/bin/forge-py .forge/lib/derived_state.py --<flag> [--format=<fmt>]`

| Flag | Stdout | Exit code |
|------|--------|-----------|
| `--detect-mode` | `split-file` / `generated` / `skip-canonical` (one line) | 0 success; â‰Ą2 on degenerate state |
| `--get-backlog --format=json\|table\|count` | JSON array / TSV table / single integer | 0 success |
| `--get-spec-index --format=json\|table\|count` | JSON array / TSV table / single integer | 0 success |
| `--get-changelog --format=json\|table\|count` | JSON array / TSV table / single integer | 0 success |
| `--skip-canonical-write` | `skip\n` (split-file) or `proceed\n` (otherwise) | 0 success; â‰Ą1 on helper error |

**Exit-code convention** (DA round-6 finding 2 â€” fixed inline at /implement): 0 always means "helper succeeded"; nonzero means "helper failed; caller MUST abort the slash command's canonical-write step". The skip/proceed decision is signaled exclusively via stdout, NOT exit code. Exit-code-based dispatch on `--skip-canonical-write` is forbidden because that would conflate "write should be skipped" with "helper failed".

### Per-command invocation table

| Command | Read flag | Format | Purpose |
|---------|-----------|--------|---------|
| `/now` | `--get-backlog` | json | Top-of-queue ranking |
| `/now` | `--get-spec-index` | json | Validation queue + stale draft scan |
| `/brainstorm` | `--get-spec-index` | count | First-run detection |
| `/brainstorm` | `--get-backlog` | json | Empty-signal check |
| `/insights` | `--get-backlog` | json | Velocity + aging |
| `/insights` | `--get-spec-index` | json | TRACKED-set construction |
| `/evolve` | `--get-backlog` | json | Closed-spec confirmation + deferred-scope aging |
| `/forge stoke` | `--detect-mode` | n/a | Migration advisory trigger |
| `/matrix` | `--get-backlog` | json | Score-integrity + status-sync inputs |
| `/matrix` | `--skip-canonical-write` | n/a | Write-side gate |
| `/spec` | `--get-spec-index` | json / count | Next-spec-number determination |
| `/spec` | `--skip-canonical-write` | n/a | Write-side gate |
| `/implement` | `--get-backlog` | json | `next` argument resolution |
| `/implement` | `--skip-canonical-write` | n/a | Write-side gate |

| `/close` | `--skip-canonical-write` | n/a | Write-side gate |

### Write-side skip rule

`/spec`, `/implement`, `/close`, `/matrix` MUST consult `.forge/bin/forge-py .forge/lib/derived_state.py --skip-canonical-write` before any canonical-table-row write to `docs/backlog.md`, `docs/specs/README.md`, or `docs/specs/CHANGELOG.md`. If stdout is `skip` (split-file mode), the canonical writes are SUPPRESSED â€” the spec frontmatter edit is the source of truth, and renderer-owned `.generated/` artifacts pick up the new state on next render. Event-stream writes (`.forge/state/events/<spec-id>/`) proceed unchanged.

If stdout is `proceed` (Phase 1 dual-write under modes `generated` or `skip-canonical`), perform the canonical writes as before.

If the helper exits nonzero, **abort the canonical-write step entirely** and surface stderr. Do NOT default to either skip or proceed â€” exit-code/skip-proceed conflation is the bug class this rule prevents.

### Diagnosis decision tree â€” "I edited docs/backlog.md and my changes disappeared"

1. **Did the edit target the operator-narrative region or the rendered-table region?**
   - Narrative (above/below the `<!-- FORGE-INCLUDE: ... -->` marker): edit persists.
   - Table region: in split-file mode the table comes from `.generated/`, not from the curated parent. Direct edits to the table region are shadowed by the include marker.
2. **What does `.forge/bin/forge-py .forge/lib/derived_state.py --detect-mode` print?**
   - `split-file`: expected behavior â€” table edits must go via spec frontmatter (the renderer reads frontmatter, not the parent file).
   - `generated`: pre-Spec-398 mode; canonical file was the rendered artifact.
   - `skip-canonical`: legacy mode; canonical file is operator-curated.
   - Helper exits â‰Ą2 with "degenerate split-file state": migration is mid-flight. Run `python scripts/migrate-to-derived-view.py --mode=split-file` or remove `docs/.generated/` to revert.
3. **Did the slash command actually run the helper, or did it open canonical files?** Run `tests/test-spec-399-read-paths.py` to assert helper-invocation presence across all 4 mirrors. If a command bypassed the helper, that is a Spec-399 regression â€” file a hotfix.
4. **Is the wrapper resolving Python correctly?** (Spec 401 territory)
   - `interpreter not found` â†’ install Python per `docs/process-kit/cross-platform-python-guide.md`.
   - `Microsoft Store stub` on Windows â†’ wrapper skips the stub automatically; if you see this error, install real Python.
   - `Python 3.10+ required (found 3.X)` â†’ upgrade Python.
   - Test wrapper presence: `for d in .forge/bin template/.forge/bin; do test -f "$d/forge-py" && test -f "$d/forge-py.cmd"; done`.
5. **Is the helper hung or slow?** The helper's CLI should complete in <2s on a typical FORGE-self repo. If slower, check filesystem responsiveness; subprocess cold-start cost on Windows is ~200-500ms per spawn (Spec 399 round-4 residual f).

### Helper-as-contract: future schema evolution

The helper IS the schema contract. Future format changes (e.g., adding a column to backlog rows, changing event-payload shape) update the helper's parsing functions + its unit tests + the renderers. **No command edits required at retirement time.** Phase 2 #3 (universal dual-write retirement) becomes a 1-line edit to `should_skip_canonical_write` (always return True).

### Constraints (inherited from Spec 401)

- Helper is **stdlib-only** â€” no PyYAML or third-party imports. Verified by `tests/test-spec-401-stdlib-only-audit.py` extended to include `derived_state.py`.
- Helper uses **binary-mode I/O** for all file reads (Spec 398 lesson â€” CRLF/BOM safety).
- Helper does NOT accept a `--project-root` CLI flag â€” root is derived from CWD only (path-traversal hardening; DA round-6 finding 8).

## Auto-migration on `copier update` (Spec 400)

Consumer projects upgrading from a pre-Spec-398 template version receive a Copier `_tasks` hook that auto-invokes `scripts/migrate-to-derived-view.py --mode=split-file`. This closes the half-migrated-state failure mode where Phase 1 dual-writes happen but the read-side rewires (Spec 399) start expecting `docs/.generated/`.

### When the hook fires

The hook fires on every `copier update` AND `copier copy` invocation, but the gate is a **sentinel-file presence test** â€” `.forge/migrations/spec-398.applied`. The migration script writes this single-line file (ISO timestamp + spec ID, e.g., `2026-05-07T23:23:11Z spec-398`) on successful completion. Once present, the gate's `test -f` returns 0 and the migration is skipped (no-op).

The hook also requires a **clean working tree** â€” `git status --porcelain` must be empty. If dirty, the hook aborts with exit code 2 and surfaces the dirty paths via stderr; the migration script is NOT invoked.

| Condition | Behavior |
|-----------|----------|
| Sentinel absent + clean tree | Migration runs; sentinel created on success |
| Sentinel absent + dirty tree | Hook aborts with exit 2; "stash or commit before copier update" |
| Sentinel present | Hook is a no-op (skipped via `test -f && exit 0`) |
| `sh` missing from PATH | Hook aborts with actionable redirect to cross-platform-python-guide |
| Migration script exits non-zero | Copier `_tasks` failure surfaces stderr verbatim and aborts the update |

### What the hook does

When the gate passes, the hook invokes:

```sh
.forge/bin/forge-py scripts/migrate-to-derived-view.py --mode=split-file
```

The Python invocation goes through the Spec 401 wrapper for cross-platform interpreter resolution + Python â‰Ą3.10 floor enforcement. The migration script:

1. Seeds per-spec event streams from CHANGELOG, activity-log, and spec frontmatter (idempotent).
2. Creates `docs/.generated/` with renderer-owned artifacts (`backlog-table.md`, `changelog-entries.md`, `spec-index-table.md`).
3. Atomically replaces each curated parent's H2-bounded rendered region with a `<!-- FORGE-INCLUDE: ... -->` marker.
4. Writes `.forge/migrations/spec-398.applied` (sentinel file).

### When the hook does NOT fire

- **Sentinel already present**: re-running `copier update` after a successful migration is a no-op. The sentinel persists; no migration logic runs.
- **Dirty working tree**: hook aborts with exit 2; migration NOT invoked. Operator runs `git stash` (or commits) and re-runs `copier update`.
- **`sh` not on PATH**: hook aborts with exit 2 + actionable guidance. Operator follows [cross-platform-python-guide.md](cross-platform-python-guide.md) Â§ Windows prerequisite detail to install Git for Windows option 3 OR launch from a Git Bash terminal.

### Manual fallback for non-Copier upgrade paths

For consumers upgrading via `git pull` / direct file sync / `forge stoke` (i.e., NOT `copier update`), Spec 399's `/forge stoke` advisory surfaces a 4-condition migration recommendation. The two surfaces are mutually exclusive: this Copier hook fires on `copier update` paths; Spec 399's advisory covers everything else. No double-prompting.

To trigger migration manually:

```sh
.forge/bin/forge-py scripts/migrate-to-derived-view.py --mode=split-file
```

Idempotent â€” running it on an already-migrated project is a no-op (sentinel-write overwrites with a fresh timestamp; renderer-owned artifacts are regenerated identically).

### Rollback procedure on failure

Copier's `_tasks` failure semantics restore tracked rendered files to their pre-update state (transactional). However, the migration script's writes are **NOT** auto-rolled back: `docs/.generated/` and `.forge/migrations/spec-398.applied` are not Copier-tracked artifacts.

To fully roll back a failed mid-flight migration:

```sh
git restore docs/backlog.md docs/specs/CHANGELOG.md docs/specs/README.md
rm -rf docs/.generated/
rm -f .forge/migrations/spec-398.applied
```

After rollback, the next `copier update` re-fires the hook (sentinel absent again).

### Sentinel file survives Copier-rolled-back update

If `copier update` rolls back due to a non-migration `_tasks` failure (e.g., the scrub step fails AFTER the migration step ran), the sentinel file persists because it lives outside the Copier-tracked file set. This is intentional â€” once the migration succeeds, re-running it should be a no-op. Operator awareness: if you need to re-run the migration after a Copier rollback, delete the sentinel manually (`rm -f .forge/migrations/spec-398.applied`) and re-run `copier update`.

### Fresh-copy paths

A `copier copy` against a post-Spec-398 template ships the curated parents pre-rendered with include markers AND `docs/.generated/` populated by the renderer's first run. The hook fires on the fresh-copy path too â€” sentinel-absent triggers migration, which idempotently confirms the existing artifacts and writes the sentinel. **First-render also marks the migration boundary** â€” both fresh-copy and upgrade-past-Spec-398 paths end with the sentinel present, ensuring uniform post-condition regardless of how the consumer arrived.
